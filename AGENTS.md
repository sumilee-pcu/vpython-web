# AGENTS.md — Codex / GPT 작업 지침 (vpython-web)

이 파일은 **OpenAI Codex(및 GPT 계열 에이전트)** 가 이 저장소에서 작업할 때 자동으로 읽는 지침이다.
Claude용 쌍둥이 문서 `CLAUDE.md`와 내용은 같지만, 여기서는 규칙을 더 명시적으로 풀어 썼다.
**둘 중 이 파일(AGENTS.md)을 기준으로 삼으면 된다.**

> ⚠️ 이 프로젝트는 문서 주도(doc-driven)로 관리된다. 코드보다 문서가 먼저다.
> 추측으로 구현하지 말고, 아래 "작업 시작 절차"를 반드시 순서대로 따를 것.

## 0. 30초 요약

- **무엇**: 브라우저에서 진짜 CPython(Pyodide/WASM)으로 VPython 물리 시뮬레이션을 돌리는 웹 앱.
  GlowScript의 대체물이되, Python→JS 트랜스파일이 아니라 실제 파이썬이라 numpy 등을 쓸 수 있는 게 차별점.
- **구조**: `에디터 → (rate()→await 변환) → Pyodide(vpython.py) → JS 브리지 globalThis.vpw → renderer.js(three.js)`
- **서버 없음**: 전부 정적 파일. 백엔드 도입 금지(클라우드 저장 Phase 8 제외).
- **현재 상태**: Phase 1 완료. 다음은 Phase 2(3D 객체 확장). 진행 상태의 진실은 `docs/ROADMAP.md`.

## 1. 작업 시작 절차 (이 순서를 건너뛰지 말 것)

1. **이 파일 전체**를 읽는다.
2. `docs/ROADMAP.md`를 읽고 **아직 `- [ ]`(미완료)인 가장 앞 항목**을 찾는다. 그것이 다음 작업이다.
3. 그 항목이 속한 단계의 상세 스펙 `docs/specs/phaseN-*.md`를 **처음부터 끝까지** 읽는다.
   여기에 API 시그니처·기본값·구현 노트·수용 기준(붙여넣어 실행 가능한 코드)이 다 있다.
4. `docs/ARCHITECTURE.md`에서 관련 절(브리지 프로토콜, 실행 모델, 회전 수식 등)을 확인한다.
5. **구현 → 검증(§5) → 스펙의 수용 기준 통과 확인 → ROADMAP 체크박스 갱신 → 커밋(§6)**.

작업 범위를 스스로 넓히지 말 것. 한 번에 한 단계, 스펙에 적힌 것만 구현한다.

## 3. 불변 규칙 (GOLDEN RULES — 어기면 기존 코드가 조용히 깨진다)

각 규칙에 **왜**와 **어기면 생기는 증상**을 붙였다. 반드시 지킬 것.

1. **브리지는 JSON 문자열만 주고받는다.**
   Python↔JS 사이로 Pyodide proxy 객체(파이썬 객체, 함수)를 직접 넘기지 말 것.
   vector는 `[x, y, z]` 배열로, 색은 `[r, g, b]`(각 0.0~1.0)로 인코딩한다.
   *어기면*: proxy 수명 관리 버그 → 무작위 크래시/메모리 누수.

2. **모든 비동기 자원은 세대(generation) 카운터로 정리한다.**
   Run을 다시 누르면 `_new_run()`이 `_generation`을 +1 한다. 실행 중이던 루프는 다음 `rate()`에서
   `_Stopped` 예외로 스스로 종료된다. 새로 만드는 이벤트 핸들러·타이머·대기 코루틴도
   **자신이 시작된 세대와 현재 세대를 비교해서** 다르면 즉시 중단해야 한다.
   *어기면*: Run을 두 번 누르면 이전 시뮬레이션이 안 죽고 겹쳐 돌아간다(유령 실행).

3. **코드 변환은 줄 수를 보존한다.**
   `rate(`→`await rate(` 치환은 `src/vpython.py`의 `_transform()`이 tokenize 기반으로 처리한다.
   줄을 추가/삭제하지 말 것 — 에러 줄번호가 사용자 코드와 어긋난다.
   *어기면*: 에러 메시지의 "N행"이 실제 코드 줄과 안 맞아 디버깅 불가.

4. **동기화 속성은 `_Object3D._known`에 있는 것만이다.**
   `pos, axis, up, color, opacity, size, radius, visible`. 여기에 새 속성을 추가하면
   `renderer.js`의 `_applyProps`에도 **반드시** 대응 처리를 추가해야 한다(짝을 맞춰라).
   그 외 속성(velocity, mass 등)은 자유 속성으로 그냥 파이썬 인스턴스 속성이 된다.
   *어기면*: 속성을 바꿔도 화면에 반영 안 됨 / 또는 JSON 직렬화 에러.

5. **서버 없는 정적 사이트를 유지한다.**
   빌드 도구(webpack/vite/tsc 등) 도입 금지. 모든 라이브러리는 CDN 또는 저장소 내 파일.
   `index.html`을 그대로 열어 GitHub Pages에 배포 가능해야 한다.
   *어기면*: 배포 파이프라인이 깨지고 프로젝트 방침 위반.

6. **`applyUpdates`/`updateObject`는 모르는 id를 조용히 무시한다.**
   이전 세대의 잔여 업데이트가 도착할 수 있기 때문. 이 `if (!mesh) continue;` 방어를 지울 것.
   *어기면*: Run 재실행 순간 null 참조 크래시.

7. **동기화 vector는 대입 시 복사한다(복사 시맨틱).**
   `b.pos = v`는 `v`를 그대로 참조하지 않고 복사본을 만들어 owner를 바인딩한다.
   같은 vector를 두 객체에 대입해도 서로 간섭하면 안 된다(VPython 동작).
   *어기면*: 한 객체를 움직였는데 다른 객체까지 따라 움직인다.

## 4. 코드 스타일

- 주변 코드의 스타일·주석 밀도·네이밍을 그대로 따른다. 새 관례를 도입하지 말 것.
- 주석은 **한국어**, "왜"를 적는다(무엇을 하는지는 코드가 말한다). 기존 파일들이 그렇게 돼 있다.
- Python은 표준 라이브러리만(Pyodide 기본 제공). 외부 pip 패키지는 Phase 7의 명시적 절차로만.
- JS는 ES 모듈 + three.js. 프레임워크 추가 금지.

## 5. 검증 방법 (Codex는 여기를 특히 주의)

**핵심 원칙: 눈으로 보는 검증에 의존하지 말고, 코드로 판정하라.**
Claude 세션과 달리 Codex 환경에는 브라우저 프리뷰 도구가 없을 수 있다. 그래서:

### 5a. 순수 로직은 브라우저 없이 검증 (권장 기본기)
vector 연산, axis↔size 결합, 코드 변환 같은 **JS 브리지와 무관한 로직**은
`from js import vpw`만 가짜(mock)로 바꾸면 로컬 CPython에서 그대로 돌릴 수 있다.
Phase 3 스펙에 이를 위한 `src/vpmath.py` 분리 + `tests/`(pytest) 계획이 있으니,
가능하면 **이 테스트 인프라를 먼저 세우고** 수용 기준을 assert로 옮겨 `python -m pytest`로 확인하라.
스펙의 수용 기준 코드 상당수가 이미 assert 형태라 이 방식과 잘 맞는다.

### 5b. 렌더링/통합은 브라우저에서 (수동 최소 절차)
```bash
cd vpython-web
python -m http.server 8000
# 브라우저에서 http://localhost:8000 접속 → 코드 붙여넣고 Run
```
확인 체크리스트:
1. Run 클릭 시 예제가 돈다.
2. 브라우저 콘솔(F12)에 에러 0건.
3. Run을 **다시** 누르면 이전 시뮬레이션이 멈추고 새로 시작한다(규칙 2 확인).
4. `examples/`의 **기존 예제 전부** 여전히 동작한다(회귀).

### 5c. 자동화(가능하면)
Playwright가 설치돼 있으면 헤드리스로 페이지를 열어 `globalThis.vpw.objects`와
출력창 텍스트를 읽어 수용 기준을 자동 판정할 수 있다(Phase 9.2 스펙 참고).
`await`가 붙은 top-level 루프는 백그라운드 탭에서 타이머 스로틀로 느려지니
헤드리스 측정 시 유의(스펙에 명시됨).

**어떤 방식이든, "구현했다"가 아니라 "해당 스펙의 수용 기준을 통과했다"를 커밋 메시지에 근거로 남길 것.**

## 6. 커밋 & 푸시

- 커밋 메시지는 **영어**, 첫 줄 = 명령형 "동사원형 + 무엇을" (예: `Add cylinder and arrow objects`).
- 기능 단위로 커밋한다. 그 기능에 딸린 spec/ROADMAP 갱신을 **같은 커밋**에 포함한다.
- 원격: `https://github.com/sumilee-pcu/vpython-web`, `main` 브랜치에 직접 푸시.
- 푸시 전 반드시 §5의 회귀 체크(기존 examples 동작)를 통과할 것.
- 커밋 꼬리표(있어도 되고 없어도 됨)는 도구가 자동으로 붙이는 것을 따른다.

## 7. 이 코드베이스에서 특히 자주 틀리는 것 (안티패턴)

Codex가 실제로 실수하기 쉬운 지점을 미리 못박아 둔다:

- ❌ **`_known`에 속성만 추가하고 renderer.js는 안 고침** → 규칙 4 위반. 항상 짝으로 수정.
- ❌ **정규식으로 rate 변환을 되돌림** → 문자열/주석 안의 `rate(`까지 바꿔 버린다.
  이미 tokenize 기반으로 해결돼 있으니 `_transform()`을 정규식으로 "단순화"하지 말 것.
- ❌ **새 async 함수(sleep, waitfor 등) 추가 시 세대 확인 누락** → 유령 실행. 규칙 2.
- ❌ **`_Object3D`를 dict `_props` 방식으로 되돌림** → 읽기 성능 5배 저하.
  현재는 의도적으로 실제 인스턴스 속성에 저장한다(ARCHITECTURE.md "속성 동기화 모델" + phase1 구현 노트).
- ❌ **pos가 "중심"인 객체와 "시작점"인 객체를 혼동** → cylinder/arrow/cone은 시작점 기준이다.
  Phase 2 스펙 표를 반드시 확인.
- ❌ **기본값을 문서 확인 없이 임의로 정함** → 스펙에 "문서 확인" 표시가 있으면
  `glowscript.org/docs/VPythonDocs/<객체>.html`에서 실제 기본값을 확인하고 넣을 것.
- ❌ **한 커밋에 여러 Phase를 몰아서 구현** → 리뷰/롤백 불가. 한 번에 한 단계.
